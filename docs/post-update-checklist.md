# 掼蛋记牌实验室 测试策略与 Post-Update Checklist

## 0. 游戏开发里应有哪些测试类型，以及如何确保项目符合要求

游戏项目不能只靠“能跑起来”判断是否可交付。当前项目虽然不是实时联机游戏，但依然有明确的规则引擎、状态机、前端交互和外部 AI 接口，因此至少要把测试拆成下面几类：

### 0.1 逻辑单元测试（Unit Test）

目标：验证纯逻辑函数和小范围状态转换是否正确。

适用范围：

- 牌面编码/解码
- 手牌展示排序
- 牌型合法性与压制关系
- 回放快照构建
- AI 决策失败时的本地回退策略

当前项目里，单元测试是“最基础的收敛条件”。只要逻辑层改动，就必须至少有相应的测试覆盖或 checklist 解释为什么暂时无法自动化。

### 0.2 状态机与流程测试

目标：验证完整用户流程中的关键状态转换是否正确。

适用范围：

- AI 模式开局不应误判结束
- 点击“上一步”后再次“下一步”应先回放已生成动作
- Challenge 触发与继续流程
- 新牌局切换时旧请求应被中止

这类测试不一定都能完全做成单元测试，但至少要在每次改动后纳入 post-update-checklist。

### 0.3 接口联通性测试

目标：验证外部依赖是否真正可用，而不是只看代码路径存在。

适用范围：

- OpenRouter API Key 连通性检测
- 目标模型是否存在
- 首次 AI 出牌请求是否能成功返回
- 401 / 429 / 网络错误是否能给出正确前端反馈

接口联通性测试通常需要“假 key + 真 key”两组思路：

- 假 key：验证错误提示是否正确
- 真 key：验证完整对战链路是否可用

### 0.4 前端交互与可视检查

目标：验证用户真正看到和点击到的界面是否可用。

适用范围：

- 字号、字距、行高是否足够可读
- 本家手牌是否能整排显示
- 像素风样式是否被破坏
- 移动端 / 窄屏下牌桌与控件是否仍可操作

这部分通常需要人工检查，必要时用浏览器自动化辅助。

### 0.5 文档一致性检查

目标：保证代码、产品文档和开发流程不脱节。

适用范围：

- README 中的功能描述与实际实现一致
- product-plan 中的设计细节与代码一致
- checklist 本身随着新需求同步更新

### 0.6 如何确保项目开发符合要求

当前项目采用“两层收敛”：

1. 自动收敛：`npm run check:update`
2. 人工收敛：执行本文件中的 post-update-checklist

只要任一项失败，就不能认为这次开发完成，必须修复后重新从 checklist 顶部执行一遍。

## 1. 执行规则

每次代码更新后，必须执行：

1. `npm run check:update`
2. 按本文件的 checklist 逐项检查

如果自动检查失败：

1. 停止提交
2. 修复代码 / 测试 / 文档
3. 重新执行 `npm run check:update`
4. 再次从本 checklist 顶部检查

如果人工检查失败：

1. 停止提交
2. 修复实现或文档
3. 重新执行对应自动检查
4. 再次从本 checklist 顶部检查

## 2. Checklist 维护规则

每次出现以下任一情况，都必须更新本 checklist：

1. 新增用户需求
2. 修改产品设计
3. 修改核心规则
4. 新增外部依赖或部署流程
5. 新增一个之前没有覆盖的缺陷类型

更新方式：

- 如果是新功能，新增对应检查项
- 如果是已存在功能的行为改变，更新原有检查项
- 如果是新发现的回归风险，新增“回归检查”项

## 3. Post-Update Checklist

### 3.1 自动检查

- [ ] `npm run test` 通过
- [ ] `npm run build` 通过
- [ ] `npm run lint` 通过

### 3.2 规则与回放逻辑

- [ ] 本地模式能正常开局、推进、回退、结束
- [ ] `buildReplaySnapshot` 在 0 动作时不会误判完成
- [ ] 已完成动作的回放快照与剩余手牌一致
- [ ] 本家手牌展示顺序为：大小王 -> 级牌 -> 其余牌降序

### 3.3 AI 对战模式

- [ ] 保存 AI 配置后能正常进入 AI 模式，不会开局直接结束
- [ ] 点击“下一步”时，浏览器 Network 面板能看到 OpenRouter 请求
- [ ] 发送给 AI 的结构化局面数据中，按轮包含领出方（leader）以及 self / partner / opponent 关系字段
- [ ] AI 完成一手出牌后，如果牌局未结束，“下一步”仍可继续点击，不会被误判为整局结束
- [ ] 若点击过“上一步”，再次点击“下一步”时先回放隐藏动作，再继续生成新动作
- [ ] AI 返回非法出牌时，系统能本地回退，不会让牌局崩溃
- [ ] AI 获取出牌信息失败时会在后台自动重试 3 次；重试仍失败时显示错误，不会直接被当作过牌
- [ ] 若 AI 出牌被本地兜底改写，本家显示的理由会同步变成兜底说明，不会和实际落地动作不一致
- [ ] API Key 无效时会显示正确错误信息
- [ ] API 连通性检测按钮可用，能反馈成功/失败状态

### 3.4 前端交互与视觉

- [ ] 主要文字在桌面、弹窗、设置页中均清晰可读
- [ ] 当前级牌信息固定显示在牌桌左上角，能一眼辨认
- [ ] 本家整手牌默认无需横向拖动即可查看完整
- [ ] AI 出牌理由只在“本家刚出牌”时显示在手牌栏，不显示在牌桌中心，也不显示其他家的理由
- [ ] 牌桌、背景、按钮、标签保持像素风和统一调色
- [ ] 720px 以下与 480px 以下断点仍可操作

### 3.5 结果与总结行为

- [ ] 每一轮结束后不会误弹“本局总结”
- [ ] “本局总结”只会在整局真正结束后自动弹出，或由奖杯按钮手动打开

### 3.6 文档与流程

- [ ] README 已反映本次行为变化
- [ ] `docs/product-plan.md` 已反映本次设计变化
- [ ] 如果需求变更影响检查标准，本文件已同步更新
- [ ] 未把任何真实 API Key 写入仓库文件

## 4. 当前自动化覆盖范围

当前已具备的单元测试覆盖：

1. 手牌展示排序规则
2. 回放快照初始状态与完成状态
3. GameManager 在 AI 领出非法/过牌时的本地回退
4. AI prompt 中包含 leader / relation 结构化轮次信息
5. AI 请求失败时后台自动重试 3 次

后续建议继续补充：

1. 牌型枚举与 `canBeat`
2. `resolveCardsFromCodes` 多副牌去重解析
3. Challenge 触发时机
4. App 中 AI 模式“回退后再前进”的组件级测试
