# 掼蛋记牌实验室 测试策略与 Post-Update Checklist

## 0. 游戏开发里应有哪些测试类型，以及如何确保项目符合要求

游戏项目不能只靠“能跑起来”判断是否可交付。当前项目虽然不是实时联机游戏，但依然有明确的规则引擎、状态机、前端交互和外部 AI 接口，因此至少要把测试拆成下面几类：

### 0.1 逻辑单元测试（Unit Test）

目标：验证纯逻辑函数和小范围状态转换是否正确。

适用范围：

- 牌面编码/解码
- 手牌展示排序
- 牌型合法性与压制关系
- 回放快照构建
- AI 决策失败时的本地回退策略

当前项目里，单元测试是“最基础的收敛条件”。只要逻辑层改动，就必须至少有相应的测试覆盖或 checklist 解释为什么暂时无法自动化。

### 0.2 状态机与流程测试

目标：验证完整用户流程中的关键状态转换是否正确。

适用范围：

- AI 模式开局不应误判结束
- 点击“上一步”后再次“下一步”应先回放已生成动作
- Challenge 触发与继续流程
- 新牌局切换时旧请求应被中止

这类测试不一定都能完全做成单元测试，但至少要在每次改动后纳入 post-update-checklist。

### 0.3 接口联通性测试

目标：验证外部依赖是否真正可用，而不是只看代码路径存在。

适用范围：

- OpenRouter API Key 连通性检测
- 目标模型是否存在
- 首次 AI 出牌请求是否能成功返回
- 401 / 429 / 网络错误是否能给出正确前端反馈

接口联通性测试通常需要“假 key + 真 key”两组思路：

- 假 key：验证错误提示是否正确
- 真 key：验证完整对战链路是否可用

### 0.4 前端交互与可视检查

目标：验证用户真正看到和点击到的界面是否可用。

适用范围：

- 字号、字距、行高是否足够可读
- 本家手牌是否能整排显示
- 像素风样式是否被破坏
- 移动端 / 窄屏下牌桌与控件是否仍可操作

这部分通常需要人工检查，必要时用浏览器自动化辅助。

### 0.5 文档一致性检查

目标：保证代码、产品文档和开发流程不脱节。

适用范围：

- README 中的功能描述与实际实现一致
- product-plan 中的设计细节与代码一致
- checklist 本身随着新需求同步更新

### 0.6 如何确保项目开发符合要求

当前项目采用“两层收敛”：

1. 自动收敛：`npm run check:update`
2. 人工收敛：执行本文件中的 post-update-checklist

只要任一项失败，就不能认为这次开发完成，必须修复后重新从 checklist 顶部执行一遍。

## 1. 执行规则

每次代码更新后，必须执行：

1. `npm run check:update`
2. 按本文件的 checklist 逐项检查

如果自动检查失败：

1. 停止提交
2. 修复代码 / 测试 / 文档
3. 重新执行 `npm run check:update`
4. 再次从本 checklist 顶部检查

如果人工检查失败：

1. 停止提交
2. 修复实现或文档
3. 重新执行对应自动检查
4. 再次从本 checklist 顶部检查

如果以上检查全部通过，且本轮存在仓库文件改动：

1. 先核对 `git status`
2. 提交本轮改动
3. push 到 `main`
4. 若 push 失败，继续排查认证或网络；只有确认无法自动处理时，才向用户说明阻塞原因

## 2. Checklist 维护规则

每次出现以下任一情况，都必须更新本 checklist：

1. 新增用户需求
2. 修改产品设计
3. 修改核心规则
4. 新增外部依赖或部署流程
5. 新增一个之前没有覆盖的缺陷类型

更新方式：

- 如果是新功能，新增对应检查项
- 如果是已存在功能的行为改变，更新原有检查项
- 如果是新发现的回归风险，新增“回归检查”项

## 3. Post-Update Checklist

### 3.1 自动检查

- [ ] `npm run test` 通过
- [ ] `npm run build` 通过
- [ ] `npm run lint` 通过

### 3.2 规则与回放逻辑

- [ ] 本地模式能正常开局、推进、回退、结束
- [ ] 策略模式能正常开局、推进、回退、结束，不会退回旧的 AI 分支或直接报错
- [ ] 策略模式下本家理由区与调试监视器会随着实际动作同步刷新，不会显示成 AI 理由或空白
- [ ] `buildReplaySnapshot` 在 0 动作时不会误判完成
- [ ] 已完成动作的回放快照与剩余手牌一致
- [ ] 本家手牌展示顺序为：大小王 -> 级牌 -> 其余牌降序

### 3.3 AI 对战模式

- [ ] 保存 AI 配置后能正常进入 AI 模式，不会开局直接结束
- [ ] 点击“下一步”时，浏览器 Network 面板能看到 OpenRouter 请求
- [ ] 连续快速点击“下一步”不会并发触发多次同手请求；同一时刻最多只会有一个推进请求在进行
- [ ] 发送给 AI 的结构化局面数据中，按轮包含 leader、seatId、team、relationToYou 等字段，且不会把界面相对称呼误当作敌我依据
- [ ] AI 完成一手出牌后，如果牌局未结束，“下一步”仍可继续点击，不会被误判为整局结束
- [ ] 若点击过“上一步”，再次点击“下一步”时先回放隐藏动作，再继续生成新动作
- [ ] AI 返回非法出牌时，系统能本地回退，不会让牌局崩溃
- [ ] AI 获取出牌信息失败时会在后台自动重试 3 次；重试仍失败时显示错误，不会直接被当作过牌
- [ ] AI 出牌请求会优先使用 structured output + response-healing；若当前模型或路由不支持，会自动回退到普通文本解析链路
- [ ] AI 当前优先从 legalActions 中返回 actionId，不再频繁出现“手牌不存在”或“牌型不合法”的协议层错误
- [ ] legalActions 会先经过本地稳健策略筛选与排序，不会把天然 4 张炸弹轻易拆成单张/对子候选去诱导 AI 误选
- [ ] 若主模型连续不按 structured output 返回，或上游 provider 限流/失败，会自动切到备用模型继续该手决策，而不是立即落回本地兜底
- [ ] AI 返回内容前即使夹带前置空行、说明文字、外层字符串、function_call.arguments 或 code fence，系统仍能提取首个有效 JSON 对象
- [ ] AI 若连续返回不可解析内容，系统会回退到本地最小合法决策，不会卡死在错误提示
- [ ] 若 AI 出牌被本地兜底改写，本家显示的理由会同步变成兜底说明，不会和实际落地动作不一致
- [ ] 调试模式下若发生格式兜底，可看到最近一次解析失败摘要，而不只是统一兜底文案
- [ ] AI prompt 只携带最近两轮结构化历史，复杂局面下不会因为长推理而频繁出现 finish_reason=length
- [ ] 牌桌中心“正在思考”文案与真实行动位一致，不会显示成当前压桌方或其他错误座位
- [ ] 调试模式开启后，可看到四家最近一次可见 AI 理由；关闭后不污染默认训练界面
- [ ] 入门难度开局会显示牌面引导，直接展示场外大小王、级牌、逢人配、A、K 的余量
- [ ] API Key 无效时会显示正确错误信息
- [ ] API 连通性检测按钮可用，能反馈成功/失败状态

### 3.4 前端交互与视觉

- [ ] 主要文字在桌面、弹窗、设置页中均清晰可读
- [ ] 当前级牌信息固定显示在牌桌左上角，能一眼辨认
- [ ] 本家整手牌默认无需横向拖动即可查看完整
- [ ] AI 出牌理由只在“本家刚出牌”时显示在手牌栏，不显示在牌桌中心，也不显示其他家的理由
- [ ] 调试模式下展示的额外 AI 理由集中在独立调试面板，不挤占牌桌中心
- [ ] 牌桌、背景、按钮、标签保持像素风和统一调色
- [ ] 720px 以下与 480px 以下断点仍可操作

### 3.5 结果与总结行为

- [ ] 每一轮结束后不会误弹“本局总结”
- [ ] “本局总结”只会在整局真正结束后自动弹出，或由奖杯按钮手动打开

### 3.6 文档与流程

- [ ] README 已反映本次行为变化
- [ ] `docs/product-plan.md` 已反映本次设计变化
- [ ] 如果需求变更影响检查标准，本文件已同步更新
- [ ] 未把任何真实 API Key 写入仓库文件
- [ ] 若本轮修改了本地出牌策略，已至少跑 20 局结构化模拟，并检查明显不合理的诊断标签
- [ ] 若本轮修改了本地出牌策略，`debug/strategy-audits/latest.json` 已更新并抽样查看关键标签样例
- [ ] 若本轮修改了结构化策略规则或策略会话，已人工验证 app 内“开始策略对局”入口和逐手推进流程

### 3.7 Challenge 教学约束

- [ ] 入门难度只会生成大牌计数题，不会再问“谁收轮 / 上一轮牌型 / 队友是谁”
- [ ] 入门难度题目会优先围绕王、级牌、A、K 与逢人配

### 3.8 会话收尾确认

- [ ] 每次结束会话前，必须向用户发起一次明确确认，并等待用户输入，以确认当前是否还有其他改进需求

## 4. 当前自动化覆盖范围

当前已具备的单元测试覆盖：

1. 手牌展示排序规则
2. 回放快照初始状态与完成状态
3. GameManager 在 AI 领出非法/过牌时的本地回退
4. AI prompt 中包含 leader / seatId / team / relationToYou 等结构化轮次信息
5. AI 请求失败时后台自动重试 3 次
6. 入门难度 challenge 只会生成计数题
7. AI 响应带前导空行、外层字符串、function_call.arguments、reasoning 或非 chat text 字段时的 JSON 提取
8. AI 从本地 legalActions 中选择 actionId 的协议稳定性

后续建议继续补充：

1. 牌型枚举与 `canBeat`
2. `resolveCardsFromCodes` 多副牌去重解析
3. Challenge 触发时机
4. App 中 AI 模式“回退后再前进”的组件级测试
