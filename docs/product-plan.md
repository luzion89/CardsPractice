# 掼蛋记牌实验室 产品与实现方案

## 1. 项目定义

这个项目不是在线对战产品，也不是规则百科，而是一个面向单局记牌训练的回放式工具。它的核心目标非常集中：

- 把一整局掼蛋拆成可逐手观察的训练材料。
- 把“记不住”变成可以反复练的回放动作和问题反馈。
- 在不引入大厅、社交、成长系统的前提下，尽量保留真实牌桌的节奏感与座位关系。

因此，项目的产品定义可以概括为：

> 预生成完整合法牌局 + 按步回放 + 按轮检索 + 局后总结。
>
> 同时支持 AI 实时对局模式：通过 OpenRouter API 驱动四家独立 AI 代理逐手出牌，生成更接近真人的牌谱。

它服务的是“训练记牌能力”的用户，而不是追求商业手游那种热闹感、奖励链路和长线留存。

## 2. 核心目标与非目标

### 2.1 核心目标

本项目当前阶段的核心目标有四个：

1. 让用户快速形成对四家座位、轮次和牌型变化的空间记忆。
2. 让用户把“看到牌”升级为“能判断场上还剩什么牌”。
3. 让训练过程可重复、可暂停、可恢复，不依赖真人凑桌。
4. 让题目和回放严格来源于同一局牌谱，避免题面与牌局脱节。

### 2.2 当前不追求的事情

以下内容不是当前阶段的重点，因此故意不做，或者只做最小化处理：

- 多局升级、进贡、抗贡、局间结算。
- 商业大厅、社交系统、排行榜、任务奖励。
- AI 绝对最优博弈。
- 赛事级复盘分析和胜率评估。

这样做不是“简化规则”，而是把复杂度集中到真正服务训练目标的地方。

## 3. 设计理念

### 3.1 产品理念

这个项目的设计理念不是“模仿一款掼蛋手游”，而是“把真实牌桌上对记牌最有价值的信息保留下来，其余噪音尽量收起来”。

具体来说：

- 主页面应该像桌面，而不是像后台。
- 牌桌永远是视觉主角，所有题目、设置、历史摘要都应该让路给牌桌。
- 用户需要感知四家相对关系，所以前端统一显示“自己 / 上家 / 下家 / 对家”，不再显示东南西北。
- 用户界面仍统一显示“自己 / 上家 / 下家 / 对家”，但 AI 内部推理必须使用 south / east / north / west 绝对座位与队伍字段，避免把界面相对称呼误当作敌我关系。
- 桌内只保留真正服务观察的元素：级牌提示和本轮出牌；剩牌数、座位标签放到牌桌外。
- 用户虽然不参与出牌，但必须能一直看到自己的本局手牌，这样才有真实视角可依附。
- 题目应该像裁判突然提问，而不是整页提示语堆叠。

### 3.2 视觉理念

当前版本采用“俱乐部牌桌 + 真实牌手视角”的方向：

- 基底是深绿大牌桌，页面主要面积都给到桌面与手牌区。
- 铜金色强调级牌、关键按钮和战报入口，营造正式但不赌博化的质感。
- 牌面采用本地 SVG 矢量生成，既保留实体牌感，也方便高亮级牌与逢人配。
- 自己的手牌固定在桌下手牌槽，随回放实时扣除，建立“我正在看自己这一手牌局”的感觉。
- 文案、统计和设置被压缩到桌外轻量区域、弹窗和局后总结，避免主界面信息污染。

这套方向有一个明确原则：

> 用户打开页面后，第一眼必须先看到桌面，不是先看到说明。

### 3.3 当前牌桌信息架构

当前版本的前端信息架构已经进一步收敛为：

- 牌桌左上角固定显示本局级牌。
- 四家座位标签和剩牌数显示在牌桌外缘，避免挤占桌面中心。
- 四家的出牌位置向桌心聚拢，用户扫一眼就能看到本轮牌权变化。
- 牌桌中间不再放大面板和 step 计数，进度条与状态文案移到牌桌下方。
- AI 出牌理由不再占用牌桌中心，只在“本家刚出牌”时显示在手牌栏里，避免桌心被解释性文案污染。
- 调试模式下额外打开独立监视面板，显示四家最近一次可见 AI 理由，用于校验 prompt 与座位理解，不进入默认训练视图。
- 桌下常驻“我的本局手牌”，让训练始终站在真实牌手视角进行。
- 入门难度在开局发牌后增加“牌面引导”，先提示王、级牌、A、K 和 10 以上缺门，帮助用户从第一步就建立观察目标。

## 4. 掼蛋产品截图参考结论

调研了公开可访问的掼蛋类截图与产品页，包括但不限于：

- 腾讯大掼蛋公开宣传图与游戏截图。
- 同城游掼蛋公开截图。
- 微乐、单机掼蛋、多乐、南京掼蛋等应用市场与下载站公开截图。

这些参考里虽然美术风格差异很大，但有几条共性非常稳定：

### 4.1 中央牌桌永远最大

商业产品再花哨，真正的可玩区域几乎都围绕一张大牌桌展开。用户头像、聊天、任务、商城、活动，都沿桌边附着，而不是挤进中心区。

### 4.2 四家信息都很短

大多数产品不会在牌桌上长篇描述玩家状态，而是只保留：

- 玩家位置
- 剩牌数
- 最近动作或当前状态

这直接支持了本项目当前的结构选择：四边信息缩成桌外短标签，桌内只保留本轮出牌，历史详情挪到弹窗。

### 4.3 关键动作应该靠牌面聚焦，而不是靠中心面板堆字

“炸弹”“不出”“头游”等信息通常会用大字、描边、发光、中心爆点来表达。用户看到的是一个高冲击的瞬时结论，而不是一整行解释文字。

这也意味着中心区域最重要的不是说明面板，而是牌本身。因此当前版本已经取消桌面中间的步骤大面板，把视觉焦点改回四家向桌心聚拢的实际出牌。

### 4.4 控件托盘化

商业产品习惯把操作按钮做成桌边托盘、底部操作坞或侧边功能带。控件不和牌面抢版心。

当前版本据此把“上一步 / 下一步 / 结束 / 新牌局”收成底部控制坞，而不是铺满页面。

## 5. 可用牌面图片资源调研结论

当前项目没有直接接入第三方整套牌面图库，而是使用自绘 SVG 牌面。这样做的主要原因是：

- 离线 PWA 场景更可靠。
- 资产尺寸可控，不需要引入整套图片资源表。
- 可直接为级牌、逢人配、大小王添加训练导向的视觉提示。

但如果后续要把牌面风格进一步向“商业平均水平”靠近，当前调研中最可用的一条路径是：

### 5.1 候选资源

- hayeah/playing-cards-assets

调研结论：

- 仓库自身采用 MIT license。
- README 明确说明牌面素材来源于 vector-playing-cards 公共领域资源。
- 同时提供 SVG 与 PNG，适合映射标准 54 张牌面。

### 5.2 为什么现在还没有直接接入

因为项目当前目标是“记牌训练”，不是“复刻商用牌桌皮肤”，所以优先级仍然是：

1. 牌谱与题目逻辑正确。
2. 牌桌与四家关系清晰。
3. 级牌、逢人配、剩余判断具备训练语义。

如果后面要进一步升级视觉真实度，完全可以把当前的 SVG 生成组件替换为静态牌面资源映射层。

## 6. 用户流程

当前产品支持两种模式：

### 6.1 AI 实时对局模式（推荐）

1. 用户进入主页，输入 OpenRouter API Key。
2. 用户点击"开始新牌局"。
3. 系统随机发牌，四家 AI 代理通过 OpenRouter API 逐手出牌。
4. 如果当前难度为入门，系统会在 0 步时先生成牌面引导，直接显示关键大牌在场外还剩多少张。
5. 用户点击"下一步"触发当前行动方的 AI 决策，实时生成牌谱。
6. 在若干推进点触发 challenge。
7. 用户答题后继续推进。
8. 牌局结束后查看战报和训练统计。
9. 用户开始下一局继续训练。

### 6.2 本地预生成模式

1. 用户进入主页，不填写 API Key，选择本地模式。
2. 系统使用启发式引擎一次性生成完整牌谱。
3. 用户一边看桌面中心的四家出牌，一边对照桌下自己的手牌。
4. 在若干推进点触发 challenge。
5. 用户答题后继续推进。
6. 牌局结束后查看战报和训练统计。
7. 用户开始下一局继续训练。

两种模式共享相同的 challenge 系统和训练统计。

这个流程的关键点是：

- 用户不需要自己出牌。
- 所有问题都必须来自当前可见历史。
- 一局结束后必须能快速进入下一局。

## 7. 规则与牌型覆盖

### 7.1 牌模型

当前引擎使用结构化的 Card 模型来表示实体牌：

- 点数 rank
- 花色 suit
- 第几副牌 deck
- 唯一 id

这样做的好处是所有训练问题都可以回到“实体牌面”统计，而不是只靠抽象点数。

### 7.2 当前支持的牌型

当前版本覆盖以下主要牌型：

- 单张
- 对子
- 三同张
- 三带二
- 顺子
- 三连对
- 钢板
- 同点炸弹
- 同花顺
- 天王炸

### 7.3 逢人配实现方式

红桃级牌在引擎里不被粗暴地当作万能牌，而是被单独识别为 wilds 池，在构造特定牌型时按需要分配到目标点数上。这样有两个好处：

- 出牌合法性可以被明确推导。
- 训练题里可以继续区分“实体牌面剩余”和“替身后的牌型含义”。

## 8. 牌局生成与引擎实现

当前项目支持两种牌局生成方式：AI 实时生成（通过 OpenRouter API）和本地启发式预生成。

### 8.0 AI 实时生成模式（OpenRouter）

在 AI 模式下，牌局不再一次性预生成，而是由 GameManager 逐手驱动四个独立的 AI 代理实时出牌：

1. 构造两副完整牌堆，共 108 张。
2. 随机选择级牌与起手座位。
3. 洗牌并给四家各发 27 张。
4. 用户每点击"下一步"，GameManager 调用当前行动方的 AIPlayerSession。
5. AI 代理通过 OpenRouter API（默认模型 minimax/minimax-m2.7）分析局面，并从本地枚举出的合法动作列表里选择一个 actionId。
6. 引擎把 actionId 映射回对应合法动作落地；若模型没有按约定返回 actionId，才退回旧的牌面编码解析与合法性校验。
7. 如果 AI 仍返回非法内容或连续不可解析，引擎自动回退到本地启发式选牌。
8. 推进轮次、更新剩牌、记录动作与收轮结果。
9. 持续进行直到牌局完成。

#### 合法动作协议

AI 与引擎之间当前优先使用“本地枚举合法动作 + AI 选择 actionId”的协议：

- 本地先根据当前手牌和压桌牌型枚举全部合法动作。
- 枚举出的动作会先经过本地稳健策略筛选与排序，优先保留不拆天然炸弹、不随意破坏顺子/连对/钢板等高价值结构的候选。
- 每个动作分配稳定的 `actionId`，并附带 `label`、`detail` 与 `cards` 供 AI 理解。
- AI 只需返回 `{"actionId":"A03","reason":"..."}`。
- 这样 AI 仍然负责策略决策，但不再负责自己组合具体牌码，从而避免“手牌不存在”或“组合不合法”这两类协议层错误。

为了兼容部分模型或路由的旧输出，系统仍保留原有牌面编码解析链路：

- 数字牌：`数字+花色字母`，如 `1a`=♠A, `5c`=♣5, `13d`=♦K
- 花色映射：a=♠, b=♥, c=♣, d=♦
- 大小王：`BJ`=大王, `SJ`=小王
- 内部 rank 14 = Ace（用于排序），编码时仍写作 1

#### AI 代理架构

每个座位拥有独立的 AIPlayerSession 实例，包含：

- 系统提示词：敌我识别、让搭档/压对手等高层策略规则，不再要求模型自己推导全部牌型合法性
- 局面提示词：当前手牌、各家剩牌数、当前压桌牌型、最近两轮结构化历史动作，以及本地枚举出的 legalActions
- 结构化轮次数据：每轮显式标注 leader、seatId、absoluteName、team、relationToYou、当前领先方，以及“是否通常应让搭档拿牌权”等上下文，帮助 AI 稳定区分敌我
- 候选动作治理：实时 AI 模式会复用本地启发式策略，把 legalActions 压缩成“更稳健的候选子集 + 更推荐的排序”，降低模型在长列表里误选拆炸/拆结构动作的概率，同时缩短 prompt 长度
- 响应解析：请求侧优先使用 structured output + response-healing 约束 JSON，并要求 provider 支持这些参数；正常路径优先解析 `actionId`，若主模型持续不按 structured output 返回，或在上游 provider 上限流/失败，会自动切到 JSON 服从度更高的备用模型重试；若上游仍不支持，再回退到兼容文本解析，从 content、text、function_call.arguments、reasoning 等位置提取旧式牌面编码和出牌理由
- 重试机制：AI 返回无效 JSON 或请求失败时，后台自动重试 3 次
- 回退机制：当 AI 明确给出非法出牌，或连续重试仍拿不到有效回复时，自动使用本地引擎给出最小合法决策；只有网络、鉴权、限流等请求类错误才直接向前端报错
- 前端串行化：`下一步` 在 UI 层增加同步锁，避免用户连续点击导致同一时刻发出多次推进请求

#### 入门开局牌面引导

当难度为入门时，系统会在发牌完成后立即给出一段“开局牌面引导”：

- 引导内容不再是解释性段落，而是直接显示场外大小王、级牌、逢人配、A、K 的剩余数量。
- 这些数量完全根据自己手牌做本地计算，避免模型措辞和计数噪音影响训练目标。
- 这段引导只出现在第 0 步，目的不是教出牌，而是让用户在第一手前先建立关键牌余量意识。

### 8.1 本地预生成模式

generateGame 的流程可以概括为：

1. 构造两副完整牌堆，共 108 张。
2. 随机选择级牌与起手座位。
3. 洗牌并给四家各发 27 张。
4. 对当前行动方分析手牌，枚举合法候选牌型。
5. 依据当前局势从候选中选出一手。
6. 推进轮次、更新剩牌、记录动作与收轮结果。
7. 持续模拟直到牌局完成。

整局在开局时一次性生成完成，所以前端的“上一步 / 下一步”只是对同一份预生成牌谱做局部解锁。

### 8.2 为什么要先预生成整局

因为训练的核心不是出牌决策，而是“观察已经发生过的历史”。预生成整局有三个直接优势：

- challenge 可以稳定从历史动作里抽题。
- 用户回退时不会导致 AI 重新洗牌或重算。
- 同一局牌谱天然可复盘、可持久化、可统计。

## 9. 四家 AI 出牌逻辑是怎么设计和实现的
当前项目有两套出牌逻辑：

### 9.0 OpenRouter AI 代理模式

在 AI 模式下，出牌由大语言模型（默认 minimax/minimax-m2.7）通过 OpenRouter API 完成。每个座位拥有独立会话，但模型当前不再直接“发明一组牌码”，而是从本地先枚举好的 legalActions 中选择 actionId。请求侧会优先尝试 structured output 与 response-healing 来压缩格式漂移；若上游不支持，再回退到多层 JSON 提取与解析。这样 AI 仍负责策略判断，但手牌存在性和牌型合法性由本地协议保证。只有在 actionId 缺失、输出持续不可解析或网络请求失败时，才进入重试或本地兜底。

这种方式的优势是既保留了“由 AI 决策”的策略风格，又通过本地候选治理显著降低了协议层非法出牌、拆炸和高破坏动作；同时 prompt 与响应都更短，能降低截断率、平均响应时延与频率限制风险。缺点仍然是依赖网络与 API 可用性，且复杂局面下仍会受到模型稳定性影响。

这一层 prompt 当前特别强调了几条掼蛋常见思路：

- 优先保留三张、连对、顺子、钢板等完整结构，不要为了走一张单牌随意拆开。
- 跟牌时能小压就小压，避免无谓用大牌抬高节奏。
- 如果当前领先者是搭档，通常不去压搭档，除非是在送搭档、自己能顺势走牌，或必须阻止对手抢回牌权。
- 对手接近走完时，提高控牌与炸弹抢权意愿；搭档接近走完时，提高送牌意愿。
- 逢人配优先用于补强关键组合，而不是轻易当普通小单牌浪费。

### 9.1 启发式本地模式（回退与本地预生成共用）

以下是本地启发式引擎的出牌逻辑，同时也作为 AI 模式的回退方案。
这一部分是项目的关键。当前不是基于神经网络，也不是基于真实对局数据库回放，而是一个“合法性优先、桌感次之”的启发式模拟器。

### 9.1 第一步：枚举当前手牌的所有合法候选

引擎会先把手牌拆解成几个视角：

- allByRank：按点数聚合所有牌。
- naturalByRank：不含逢人配的自然牌。
- sequenceByRank：可参与顺子相关牌型的牌。
- flushBySuit：按花色聚合，供同花顺使用。
- wilds：红桃级牌组成的替身池。

在这个基础上，分别生成：

- 单张、对子、三同张
- 三带二
- 顺子、三连对、钢板
- 同点炸、同花顺、天王炸

再去重，得到当前手牌可出的所有合法牌型集合。

### 9.2 第二步：领出时怎么选牌

领出使用的是 chooseLeadPlay。它不是简单地选最小牌，也不是一味追求牌型大，而是通过 scoreLead 做启发式评分。

当前评分倾向大致是：

- 优先较轻的单张、对子，减少无谓拆结构成本。
- 对顺子、三连对、钢板、三带二施加额外惩罚，避免一上来就把高价值结构打光。
- 对炸弹保留更强，特别是手牌还很多时更不愿意轻炸。
- 对高权值牌、级牌、替身牌进行额外考量。
- 如果某手能直接出完，则会获得最强优先级。

直观上看，这意味着 AI 的领出逻辑更像“先把牌路铺顺”，而不是“每步都追求理论最优”。

### 9.3 第三步：跟牌时怎么选牌

跟牌使用 chooseResponsePlay，逻辑比领出更强调局势：

- 先筛出所有能压住当前牌的候选。
- 如果队友当前领先，默认不抢；只有在自己能一手出完时才会反超。
- 如果当前压桌者剩牌很少，会进入 enemyDanger 状态，提高压制强度。
- 如果搭档已经接近走完，会提高“先把牌权抢回来再喂搭档”的意愿。
- 优先尝试同类型压制，而不是直接炸。
- 炸弹只在残局危险、手牌不多或者能直接出完时才更愿意使用。

这个策略的目的，不是为了成为强 AI，而是让回放里的节奏更接近人类牌桌：

- 有时放队友过。
- 有时小压一手。
- 不会每次都机械地用最大解。

### 9.4 第四步：轮次与收轮处理

每一轮里，引擎都会记录：

- 领出者
- 当前压住方
- 每步动作索引
- 收轮者

当全桌无人继续压住时，由当前压桌者收轮，并获得下一轮主动权；若其已经走完，则按剩余活跃座位继续选择下一个领出者。

### 9.5 为什么这种逻辑适合当前项目

因为本项目真正需要的是：

- 合法的出牌顺序
- 说得通的轮次变化
- 可被用户记忆和提问的动作历史

而不是“这手牌是不是国手级最优解”。

## 10. 题库系统如何依赖牌谱

challenge 系统并不是随机独立题库，而是严格从当前可见历史里抽题。当前版本也不再采用“低概率随机弹题”，而是把 challenge 改成“每完成一轮就暂停一次的检索练习”。其好处是：

- 不会问用户尚未看到的信息。
- 每道题都可以回到具体动作给解释。
- 题目天然和当前局势绑定，训练更像牌桌上被当场追问。
- 用户不会再出现“看了很多步却一次也没被 challenge”的失灵感。

### 10.1 记忆方法调研结论

这次对训练机制的调整，主要参考了三类较稳定的记忆研究结论：

- 主动检索优于被动重看。相比单纯继续看牌，停下来回忆“刚才是谁收轮、哪张大牌已经出现”更能强化保持。
- 分散提取优于一次性回想。记牌不是在局末一次总复盘，而是要在对局中反复把信息从脑中提出来。
- 轮次天然适合 chunking。掼蛋记牌本来就是按“这一轮谁领、谁压、谁收”来组织信息，按轮切块比按单手随机提问更符合工作记忆负担。

因此，这个产品当前采用的不是“随机小测”，而是“以轮为单位的低负担高频检索”。这比纯随机更适合记牌训练，因为它稳定、可预期，而且不会把训练价值交给概率。

### 10.2 当前采用的训练机制

当前 challenge 机制分成三层：

- 立即回忆：每轮刚结束就追问上一轮谁收轮、哪家出了什么牌型。
- 延迟回忆：在更高难度下，不只问刚结束的一轮，还会回问前两轮甚至更早的关键动作。
- 全局计数：穿插询问 A、级牌、大小王、逢人配或任意点数牌按实体牌面还剩多少。

这三层分别对应三种训练目标：

- 先把“我看到了什么”记住。
- 再把“我几轮前看到过什么”重新提取出来。
- 最后把“我对全局牌面还剩多少”从模糊印象提升成明确判断。

### 10.3 为什么是“每轮必答”而不是随机弹题

如果 challenge 只按概率触发，即使总体期望次数不低，用户仍然会因为短期内连续不触发而觉得系统没有在训练记牌。对训练产品来说，这是体验上的根本问题。

当前版本改为每轮结束后固定暂停一次，有几个直接收益：

- 节奏稳定，用户知道自己每轮都要回忆一次。
- 轮次就是天然分块，不会在奇怪的步点打断注意力。
- 更容易做难度升级：入门问即时回忆，基础混入计数，进阶加入延迟回忆与搭档判断。

### 10.4 当前难度分层

- 入门：每轮必答，主要在“上一轮谁收轮、哪家最后出了什么”。
- 基础：每轮必答，并轮换“上一轮回忆 / 大牌计数 / 最近两轮延迟回忆”。
- 进阶：每轮必答，混合“上一轮回忆 / 全局计数 / 更长延迟回忆 / 搭档关系判断”。

## 11. 当前策略与真实对局记录的差异评估

这一部分非常重要，因为它决定未来是否要引入真实牌谱数据。

### 11.1 当前启发式策略的优势

对当前项目目标来说，启发式生成有几个明显优势：

#### A. 实现成本可控

只要规则引擎写对，就能无限生成合法牌谱，不需要先建立数据采集、清洗、脱敏和标注链路。

#### B. 可解释性强

每个动作都是本地逻辑算出来的。出了什么牌、为什么能出、为什么收轮，都可以沿代码直接追溯。

#### C. 非常适合训练题生成

因为引擎掌握整局所有实体牌面，统计题和剩余牌问题可以直接计算，不依赖外部标注。

#### D. 无限可扩展

只要给新 seed，就能得到新牌局，天然适合记牌训练这种高重复需求场景。

### 11.2 当前启发式策略的局限

它也有明确缺点：

#### A. 不够像真实玩家

虽然当前逻辑已经模拟了让队友、残局压制、保炸等倾向，但它仍然缺少很多真人桌上的细腻判断，例如：

- 更激进或更保守的地域风格差异
- 针对特定搭档配合的默契打法
- 为了长线升级收益而做的局间博弈
- 人类常见的非理性选择和失误

这次调研公开规则与策略资料后，可以把“更接近现实”的方向进一步说清楚。比较稳定的真人桌倾向至少包括：

- 牌权本身很值钱，能不能收轮、能不能重新领出，经常比单手压住更重要。
- 这是搭档游戏，不是纯个人 shedding；弱牌手往往会牺牲自己，去保强搭档先走。
- 低单张、低对子、低三张有时不是“弱出牌”，而是在试探是否能用高张同型牌再夺回牌权。
- 管子、钢板这类结构型牌常常带有“逼炸”意味，不只是单纯甩牌。

当前引擎已经覆盖了其中一部分，比如保炸、让队友和残局压制；本次实现又补了一条更接近真人的判断：

- 当搭档只剩很少牌时，AI 会更愿意把牌权抢回来，包括更积极地同型压制或在必要时动用炸弹，目的是给搭档创造下一轮主动权。

#### B. 牌谱分布会带有引擎偏好

如果长期只用一种启发式生成牌谱，用户学到的会是“这套引擎常怎么打”，而不是真实人群的完整分布。

#### C. 高级战术题空间有限

当前项目适合问“还剩几张 A”“谁收轮”“上一轮哪家出了什么”，但不太适合问“为什么这一步不炸”“这手让牌是否最佳”。因为这类题需要更强的人类战术语境。

### 11.3 使用真实对局记录的优势

如果未来引入真实牌谱数据，最大的提升是“真实性”：

- 出牌节奏更像真人。
- 错误、犹豫、保守与冒进都会出现。
- 题目可以更贴近实战，例如“这一手为什么没拆炸”“这一轮为什么放队友”。
- 可以基于真实频率设计更合理的难度曲线。

### 11.4 使用真实对局记录的代价

但真实牌谱并不是免费升级，它会引入很重的实现成本：

#### A. 数据来源难题

很多公开记录只包含动作，不包含完整手牌；而记牌训练要问剩余牌，通常需要完整实体牌面。

#### B. 规则版本不统一

不同平台、不同地区、不同房间配置，对掼蛋细节规则可能有差异。真实数据需要先统一口径，才能进入训练系统。

#### C. 隐私与版权风险

商业平台牌谱往往不是拿来就能直接用，涉及抓取、存储、展示与再利用边界。

#### D. 题目生成会变复杂

真实记录如果信息不完整，很多题目就无法直接从日志里算出来，需要额外推断或放弃。

### 11.5 对本项目核心目标的实际影响

如果把目标限定为“单局记牌训练”，当前启发式策略已经能满足大部分核心需求：

- 牌谱合法
- 节奏可读
- 题目可生成
- 无限新局

如果把目标升级为“更像真人实战复盘器”，那真实对局记录会明显更好。

当前版本已经引入了 OpenRouter AI 代理模式作为中间方案：通过大语言模型驱动出牌，在不需要真实牌谱数据的前提下，获得比纯启发式更接近人类思路的出牌决策。这为未来进一步引入真实牌谱留了演进空间。

所以当前最合理的判断是：

> 启发式牌谱作为离线底座保证可用性；AI 代理模式作为 V1.5 的智能增强层提供更真实的出牌风格；真实对局记录更适合作为 V2/V3 的真实性增强层。

### 11.6 建议的演进路线

如果后续要继续做深，建议不是“全量替换”，而是混合方案：

1. 继续保留启发式生成，保证无限牌局和完整可控数据。
2. 新增少量真实牌谱专题训练，作为高阶模式。
3. 把真实牌谱用于校准启发式权重，而不是直接废掉现有引擎。

这样既能保住当前项目最强的可控性，也能逐步补足真实性。

## 12. 当前技术结构

### 12.1 前端

- React 19
- TypeScript
- Vite
- PWA manifest + service worker

### 12.2 引擎职责分层

- types.ts：定义牌、动作、轮次、快照、题目、AIConfig 等数据结构。
- engine.ts：规则枚举、牌局生成、回放快照、剩余牌统计、启发式 AI（本地模式与回退）。
- cardCode.ts：牌面编码与解码，用于 AI 通信（1a=♠A, BJ=大王 等）。
- aiService.ts：OpenRouter API 客户端，每座位独立 AIPlayerSession，含系统提示词、局面构造、响应解析。
- gameManager.ts：AI 模式下的实时对局编排器，逐手调用 AI、验证合法性、推进轮次。
- challenges.ts：依据当前可见历史抽取问题。

### 12.3 前端页面职责

- App.tsx：主桌面、弹窗、控制、战报。
- PlayingCard.tsx：本地 SVG 牌面渲染。
- App.css / index.css：牌桌视觉、控制坞、弹窗和整体主题。

### 12.4 关键交互细节（本次修订）

#### A. AI 调用链路

- 用户在登录页填写 API Key 后启动 AI 对局。
- 每次点击“下一步”时，前端通过 GameManager 调用当前行动方 AIPlayerSession。
- AIPlayerSession 通过 OpenRouter `chat/completions` 请求获取出牌决定。
- 返回结果先经过本地规则验证（手牌包含、牌型合法、能否压过）；不合法则回退到本地策略。

这意味着在浏览器 Network 面板中，AI 模式推进步骤时可以看到 OpenRouter 请求。

本次还修正了两个 AI 模式状态细节：

- 空动作开局不再被误判为“本局已结束”。
- 当用户在 AI 模式下点击“上一步”后，再点“下一步”会先回放已生成但被隐藏的动作，只有回放追平后才会发起新的 OpenRouter 请求。
- AI 请求失败或回复无效时，前端不会直接把这一步记为“过牌”，而是后台自动重试 3 次。
- Challenge 结束后不会因为“当前已显示动作数等于已生成动作数”而误弹整局总结，整局总结只在整局结束时自动出现。

#### B. API 连通性检测

- 登录页与设置页都提供“检测连通性”按钮。
- 检测会调用 OpenRouter `models` 接口，反馈 API Key 是否可用，以及目标模型是否可见。

#### C. 本家手牌展示排序

本家手牌展示不再做自动理牌分组，而是采用固定排序，从左到右：

1. 大小王
2. 级牌
3. 其余牌（从大到小）

该排序仅作用于前端展示，避免影响规则判定与出牌合法性逻辑。

#### D. 本家手牌展示布局

本家手牌不再使用“自动理牌分组”展示，但仍保留类似实体牌桌的重叠排布：

- 所有手牌在同一排内重叠铺开。
- 前端会根据容器宽度自动计算叠放步进，尽量保证用户不需要横向滚动就能看到整手牌。
- 在窄屏下会同步缩小牌面尺寸与重叠间距，优先保证整体可见。

#### E. 可读性与像素风修订

为了避免文字过小、字距过密，本次同步调整了：

- 全局字体改为更易读的正文中文字体，并单独保留像素风展示字体用于标题、标签、按钮和数字区域。
- 全局字号、行高、字距整体上调。
- 页面背景、牌桌纹理、边框、阴影和控件样式改为更明确的简约像素风。
- 级牌信息改为牌桌左上角固定牌匾，提升扫视可见性。
- 本家 AI 理由改为手牌栏内显示，只在本家出牌后可见。

### 12.5 测试与收敛方式

当前项目不再只依赖人工点点看，而是采用“双层收敛”：

1. 自动检查：`npm run check:update`，顺序执行 `test -> build -> lint`
2. 人工检查：按 `docs/post-update-checklist.md` 逐项检查 AI 模式、前端交互、文档一致性和回归风险

其中单元测试当前已覆盖：

- 本家手牌展示排序
- 回放快照初始状态与完成态
- GameManager 在 AI 领出非法 / 过牌时的本地回退
- AI prompt 中 leader / relation 的结构化轮次信息
- AI 请求失败后的 3 次自动重试

规则要求是：

- 任一自动检查失败，都必须修复后重新执行 `npm run check:update`
- 任一人工 checklist 失败，也必须修复并重新走完整检查流程
- 每次有新需求或设计变更，都要同步更新 `docs/post-update-checklist.md`

## 13. 部署方式

项目通过 GitHub Actions 在 push 到 main 后自动构建并部署到 GitHub Pages。Vite 的 base 会根据 GitHub Actions 环境自动切换，因此本地开发与 Pages 子路径部署可以共用同一套配置。

### 13.1 使用 Git Credential Manager 同步到 GitHub

为避免 HTTPS 推送时重复认证，推荐在本机配置 Git Credential Manager（GCM）：

1. `git config --global credential.helper manager-core`
2. `git credential-manager configure`

可通过下列命令确认 helper 已生效：

- `git config --global credential.helper`

日常同步流程：

1. `git add -A`
2. `git commit -m "commit message"`
3. `git push origin main`

首次 push 时，GCM 会触发 GitHub 登录授权（浏览器或设备码）；授权后本机会缓存凭据，后续可直接推送并触发 GitHub Actions 自动部署。
