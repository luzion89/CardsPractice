# 掼蛋记牌实验室 产品与实现方案

## 1. 项目定义

这个项目不是在线对战产品，也不是规则百科，而是一个面向单局记牌训练的回放式工具。它的核心目标非常集中：

- 把一整局掼蛋拆成可逐手观察的训练材料。
- 把“记不住”变成可以反复练的回放动作和问题反馈。
- 在不引入大厅、社交、成长系统的前提下，尽量保留真实牌桌的节奏感与座位关系。

因此，项目的产品定义可以概括为：

> 预生成完整合法牌局 + 按步回放 + 按轮检索 + 局后总结。

它服务的是“训练记牌能力”的用户，而不是追求商业手游那种热闹感、奖励链路和长线留存。

## 2. 核心目标与非目标

### 2.1 核心目标

本项目当前阶段的核心目标有四个：

1. 让用户快速形成对四家座位、轮次和牌型变化的空间记忆。
2. 让用户把“看到牌”升级为“能判断场上还剩什么牌”。
3. 让训练过程可重复、可暂停、可恢复，不依赖真人凑桌。
4. 让题目和回放严格来源于同一局牌谱，避免题面与牌局脱节。

### 2.2 当前不追求的事情

以下内容不是当前阶段的重点，因此故意不做，或者只做最小化处理：

- 多局升级、进贡、抗贡、局间结算。
- 商业大厅、社交系统、排行榜、任务奖励。
- AI 绝对最优博弈。
- 赛事级复盘分析和胜率评估。

这样做不是“简化规则”，而是把复杂度集中到真正服务训练目标的地方。

## 3. 设计理念

### 3.1 产品理念

这个项目的设计理念不是“模仿一款掼蛋手游”，而是“把真实牌桌上对记牌最有价值的信息保留下来，其余噪音尽量收起来”。

具体来说：

- 主页面应该像桌面，而不是像后台。
- 牌桌永远是视觉主角，所有题目、设置、历史摘要都应该让路给牌桌。
- 用户需要感知四家相对关系，所以前端统一显示“自己 / 上家 / 下家 / 对家”，不再显示东南西北。
- 桌内只保留真正服务观察的元素：级牌提示和本轮出牌；剩牌数、座位标签放到牌桌外。
- 用户虽然不参与出牌，但必须能一直看到自己的本局手牌，这样才有真实视角可依附。
- 题目应该像裁判突然提问，而不是整页提示语堆叠。

### 3.2 视觉理念

当前版本采用“俱乐部牌桌 + 真实牌手视角”的方向：

- 基底是深绿大牌桌，页面主要面积都给到桌面与手牌区。
- 铜金色强调级牌、关键按钮和战报入口，营造正式但不赌博化的质感。
- 牌面采用本地 SVG 矢量生成，既保留实体牌感，也方便高亮级牌与逢人配。
- 自己的手牌固定在桌下手牌槽，随回放实时扣除，建立“我正在看自己这一手牌局”的感觉。
- 文案、统计和设置被压缩到桌外轻量区域、弹窗和局后总结，避免主界面信息污染。

这套方向有一个明确原则：

> 用户打开页面后，第一眼必须先看到桌面，不是先看到说明。

### 3.3 当前牌桌信息架构

当前版本的前端信息架构已经进一步收敛为：

- 牌桌左上角固定显示本局级牌。
- 四家座位标签和剩牌数显示在牌桌外缘，避免挤占桌面中心。
- 四家的出牌位置向桌心聚拢，用户扫一眼就能看到本轮牌权变化。
- 牌桌中间不再放大面板和 step 计数，进度条与状态文案移到牌桌下方。
- 桌下常驻“我的本局手牌”，让训练始终站在真实牌手视角进行。

## 4. 掼蛋产品截图参考结论

调研了公开可访问的掼蛋类截图与产品页，包括但不限于：

- 腾讯大掼蛋公开宣传图与游戏截图。
- 同城游掼蛋公开截图。
- 微乐、单机掼蛋、多乐、南京掼蛋等应用市场与下载站公开截图。

这些参考里虽然美术风格差异很大，但有几条共性非常稳定：

### 4.1 中央牌桌永远最大

商业产品再花哨，真正的可玩区域几乎都围绕一张大牌桌展开。用户头像、聊天、任务、商城、活动，都沿桌边附着，而不是挤进中心区。

### 4.2 四家信息都很短

大多数产品不会在牌桌上长篇描述玩家状态，而是只保留：

- 玩家位置
- 剩牌数
- 最近动作或当前状态

这直接支持了本项目当前的结构选择：四边信息缩成桌外短标签，桌内只保留本轮出牌，历史详情挪到弹窗。

### 4.3 关键动作应该靠牌面聚焦，而不是靠中心面板堆字

“炸弹”“不出”“头游”等信息通常会用大字、描边、发光、中心爆点来表达。用户看到的是一个高冲击的瞬时结论，而不是一整行解释文字。

这也意味着中心区域最重要的不是说明面板，而是牌本身。因此当前版本已经取消桌面中间的步骤大面板，把视觉焦点改回四家向桌心聚拢的实际出牌。

### 4.4 控件托盘化

商业产品习惯把操作按钮做成桌边托盘、底部操作坞或侧边功能带。控件不和牌面抢版心。

当前版本据此把“上一步 / 下一步 / 结束 / 新牌局”收成底部控制坞，而不是铺满页面。

## 5. 可用牌面图片资源调研结论

当前项目没有直接接入第三方整套牌面图库，而是使用自绘 SVG 牌面。这样做的主要原因是：

- 离线 PWA 场景更可靠。
- 资产尺寸可控，不需要引入整套图片资源表。
- 可直接为级牌、逢人配、大小王添加训练导向的视觉提示。

但如果后续要把牌面风格进一步向“商业平均水平”靠近，当前调研中最可用的一条路径是：

### 5.1 候选资源

- hayeah/playing-cards-assets

调研结论：

- 仓库自身采用 MIT license。
- README 明确说明牌面素材来源于 vector-playing-cards 公共领域资源。
- 同时提供 SVG 与 PNG，适合映射标准 54 张牌面。

### 5.2 为什么现在还没有直接接入

因为项目当前目标是“记牌训练”，不是“复刻商用牌桌皮肤”，所以优先级仍然是：

1. 牌谱与题目逻辑正确。
2. 牌桌与四家关系清晰。
3. 级牌、逢人配、剩余判断具备训练语义。

如果后面要进一步升级视觉真实度，完全可以把当前的 SVG 生成组件替换为静态牌面资源映射层。

## 6. 用户流程

当前产品流程是单线型的：

1. 用户进入主页。
2. 系统随机生成一局完整牌谱。
3. 用户一边看桌面中心的四家出牌，一边对照桌下自己的手牌。
4. 在若干推进点触发 challenge。
5. 用户答题后继续推进。
6. 牌局结束后查看战报和训练统计。
7. 用户开始下一局继续训练。

这个流程的关键点是：

- 用户不需要自己出牌。
- 所有问题都必须来自当前可见历史。
- 一局结束后必须能快速进入下一局。

## 7. 规则与牌型覆盖

### 7.1 牌模型

当前引擎使用结构化的 Card 模型来表示实体牌：

- 点数 rank
- 花色 suit
- 第几副牌 deck
- 唯一 id

这样做的好处是所有训练问题都可以回到“实体牌面”统计，而不是只靠抽象点数。

### 7.2 当前支持的牌型

当前版本覆盖以下主要牌型：

- 单张
- 对子
- 三同张
- 三带二
- 顺子
- 三连对
- 钢板
- 同点炸弹
- 同花顺
- 天王炸

### 7.3 逢人配实现方式

红桃级牌在引擎里不被粗暴地当作万能牌，而是被单独识别为 wilds 池，在构造特定牌型时按需要分配到目标点数上。这样有两个好处：

- 出牌合法性可以被明确推导。
- 训练题里可以继续区分“实体牌面剩余”和“替身后的牌型含义”。

## 8. 牌局生成与引擎实现

### 8.1 对局生成总体流程

generateGame 的流程可以概括为：

1. 构造两副完整牌堆，共 108 张。
2. 随机选择级牌与起手座位。
3. 洗牌并给四家各发 27 张。
4. 对当前行动方分析手牌，枚举合法候选牌型。
5. 依据当前局势从候选中选出一手。
6. 推进轮次、更新剩牌、记录动作与收轮结果。
7. 持续模拟直到牌局完成。

整局在开局时一次性生成完成，所以前端的“上一步 / 下一步”只是对同一份预生成牌谱做局部解锁。

### 8.2 为什么要先预生成整局

因为训练的核心不是出牌决策，而是“观察已经发生过的历史”。预生成整局有三个直接优势：

- challenge 可以稳定从历史动作里抽题。
- 用户回退时不会导致 AI 重新洗牌或重算。
- 同一局牌谱天然可复盘、可持久化、可统计。

## 9. 四家 AI 出牌逻辑是怎么设计和实现的

这一部分是项目的关键。当前不是基于神经网络，也不是基于真实对局数据库回放，而是一个“合法性优先、桌感次之”的启发式模拟器。

### 9.1 第一步：枚举当前手牌的所有合法候选

引擎会先把手牌拆解成几个视角：

- allByRank：按点数聚合所有牌。
- naturalByRank：不含逢人配的自然牌。
- sequenceByRank：可参与顺子相关牌型的牌。
- flushBySuit：按花色聚合，供同花顺使用。
- wilds：红桃级牌组成的替身池。

在这个基础上，分别生成：

- 单张、对子、三同张
- 三带二
- 顺子、三连对、钢板
- 同点炸、同花顺、天王炸

再去重，得到当前手牌可出的所有合法牌型集合。

### 9.2 第二步：领出时怎么选牌

领出使用的是 chooseLeadPlay。它不是简单地选最小牌，也不是一味追求牌型大，而是通过 scoreLead 做启发式评分。

当前评分倾向大致是：

- 优先较轻的单张、对子，减少无谓拆结构成本。
- 对顺子、三连对、钢板、三带二施加额外惩罚，避免一上来就把高价值结构打光。
- 对炸弹保留更强，特别是手牌还很多时更不愿意轻炸。
- 对高权值牌、级牌、替身牌进行额外考量。
- 如果某手能直接出完，则会获得最强优先级。

直观上看，这意味着 AI 的领出逻辑更像“先把牌路铺顺”，而不是“每步都追求理论最优”。

### 9.3 第三步：跟牌时怎么选牌

跟牌使用 chooseResponsePlay，逻辑比领出更强调局势：

- 先筛出所有能压住当前牌的候选。
- 如果队友当前领先，默认不抢；只有在自己能一手出完时才会反超。
- 如果当前压桌者剩牌很少，会进入 enemyDanger 状态，提高压制强度。
- 如果搭档已经接近走完，会提高“先把牌权抢回来再喂搭档”的意愿。
- 优先尝试同类型压制，而不是直接炸。
- 炸弹只在残局危险、手牌不多或者能直接出完时才更愿意使用。

这个策略的目的，不是为了成为强 AI，而是让回放里的节奏更接近人类牌桌：

- 有时放队友过。
- 有时小压一手。
- 不会每次都机械地用最大解。

### 9.4 第四步：轮次与收轮处理

每一轮里，引擎都会记录：

- 领出者
- 当前压住方
- 每步动作索引
- 收轮者

当全桌无人继续压住时，由当前压桌者收轮，并获得下一轮主动权；若其已经走完，则按剩余活跃座位继续选择下一个领出者。

### 9.5 为什么这种逻辑适合当前项目

因为本项目真正需要的是：

- 合法的出牌顺序
- 说得通的轮次变化
- 可被用户记忆和提问的动作历史

而不是“这手牌是不是国手级最优解”。

## 10. 题库系统如何依赖牌谱

challenge 系统并不是随机独立题库，而是严格从当前可见历史里抽题。当前版本也不再采用“低概率随机弹题”，而是把 challenge 改成“每完成一轮就暂停一次的检索练习”。其好处是：

- 不会问用户尚未看到的信息。
- 每道题都可以回到具体动作给解释。
- 题目天然和当前局势绑定，训练更像牌桌上被当场追问。
- 用户不会再出现“看了很多步却一次也没被 challenge”的失灵感。

### 10.1 记忆方法调研结论

这次对训练机制的调整，主要参考了三类较稳定的记忆研究结论：

- 主动检索优于被动重看。相比单纯继续看牌，停下来回忆“刚才是谁收轮、哪张大牌已经出现”更能强化保持。
- 分散提取优于一次性回想。记牌不是在局末一次总复盘，而是要在对局中反复把信息从脑中提出来。
- 轮次天然适合 chunking。掼蛋记牌本来就是按“这一轮谁领、谁压、谁收”来组织信息，按轮切块比按单手随机提问更符合工作记忆负担。

因此，这个产品当前采用的不是“随机小测”，而是“以轮为单位的低负担高频检索”。这比纯随机更适合记牌训练，因为它稳定、可预期，而且不会把训练价值交给概率。

### 10.2 当前采用的训练机制

当前 challenge 机制分成三层：

- 立即回忆：每轮刚结束就追问上一轮谁收轮、哪家出了什么牌型。
- 延迟回忆：在更高难度下，不只问刚结束的一轮，还会回问前两轮甚至更早的关键动作。
- 全局计数：穿插询问 A、级牌、大小王、逢人配或任意点数牌按实体牌面还剩多少。

这三层分别对应三种训练目标：

- 先把“我看到了什么”记住。
- 再把“我几轮前看到过什么”重新提取出来。
- 最后把“我对全局牌面还剩多少”从模糊印象提升成明确判断。

### 10.3 为什么是“每轮必答”而不是随机弹题

如果 challenge 只按概率触发，即使总体期望次数不低，用户仍然会因为短期内连续不触发而觉得系统没有在训练记牌。对训练产品来说，这是体验上的根本问题。

当前版本改为每轮结束后固定暂停一次，有几个直接收益：

- 节奏稳定，用户知道自己每轮都要回忆一次。
- 轮次就是天然分块，不会在奇怪的步点打断注意力。
- 更容易做难度升级：入门问即时回忆，基础混入计数，进阶加入延迟回忆与搭档判断。

### 10.4 当前难度分层

- 入门：每轮必答，主要在“上一轮谁收轮、哪家最后出了什么”。
- 基础：每轮必答，并轮换“上一轮回忆 / 大牌计数 / 最近两轮延迟回忆”。
- 进阶：每轮必答，混合“上一轮回忆 / 全局计数 / 更长延迟回忆 / 搭档关系判断”。

## 11. 当前策略与真实对局记录的差异评估

这一部分非常重要，因为它决定未来是否要引入真实牌谱数据。

### 11.1 当前启发式策略的优势

对当前项目目标来说，启发式生成有几个明显优势：

#### A. 实现成本可控

只要规则引擎写对，就能无限生成合法牌谱，不需要先建立数据采集、清洗、脱敏和标注链路。

#### B. 可解释性强

每个动作都是本地逻辑算出来的。出了什么牌、为什么能出、为什么收轮，都可以沿代码直接追溯。

#### C. 非常适合训练题生成

因为引擎掌握整局所有实体牌面，统计题和剩余牌问题可以直接计算，不依赖外部标注。

#### D. 无限可扩展

只要给新 seed，就能得到新牌局，天然适合记牌训练这种高重复需求场景。

### 11.2 当前启发式策略的局限

它也有明确缺点：

#### A. 不够像真实玩家

虽然当前逻辑已经模拟了让队友、残局压制、保炸等倾向，但它仍然缺少很多真人桌上的细腻判断，例如：

- 更激进或更保守的地域风格差异
- 针对特定搭档配合的默契打法
- 为了长线升级收益而做的局间博弈
- 人类常见的非理性选择和失误

这次调研公开规则与策略资料后，可以把“更接近现实”的方向进一步说清楚。比较稳定的真人桌倾向至少包括：

- 牌权本身很值钱，能不能收轮、能不能重新领出，经常比单手压住更重要。
- 这是搭档游戏，不是纯个人 shedding；弱牌手往往会牺牲自己，去保强搭档先走。
- 低单张、低对子、低三张有时不是“弱出牌”，而是在试探是否能用高张同型牌再夺回牌权。
- 管子、钢板这类结构型牌常常带有“逼炸”意味，不只是单纯甩牌。

当前引擎已经覆盖了其中一部分，比如保炸、让队友和残局压制；本次实现又补了一条更接近真人的判断：

- 当搭档只剩很少牌时，AI 会更愿意把牌权抢回来，包括更积极地同型压制或在必要时动用炸弹，目的是给搭档创造下一轮主动权。

#### B. 牌谱分布会带有引擎偏好

如果长期只用一种启发式生成牌谱，用户学到的会是“这套引擎常怎么打”，而不是真实人群的完整分布。

#### C. 高级战术题空间有限

当前项目适合问“还剩几张 A”“谁收轮”“上一轮哪家出了什么”，但不太适合问“为什么这一步不炸”“这手让牌是否最佳”。因为这类题需要更强的人类战术语境。

### 11.3 使用真实对局记录的优势

如果未来引入真实牌谱数据，最大的提升是“真实性”：

- 出牌节奏更像真人。
- 错误、犹豫、保守与冒进都会出现。
- 题目可以更贴近实战，例如“这一手为什么没拆炸”“这一轮为什么放队友”。
- 可以基于真实频率设计更合理的难度曲线。

### 11.4 使用真实对局记录的代价

但真实牌谱并不是免费升级，它会引入很重的实现成本：

#### A. 数据来源难题

很多公开记录只包含动作，不包含完整手牌；而记牌训练要问剩余牌，通常需要完整实体牌面。

#### B. 规则版本不统一

不同平台、不同地区、不同房间配置，对掼蛋细节规则可能有差异。真实数据需要先统一口径，才能进入训练系统。

#### C. 隐私与版权风险

商业平台牌谱往往不是拿来就能直接用，涉及抓取、存储、展示与再利用边界。

#### D. 题目生成会变复杂

真实记录如果信息不完整，很多题目就无法直接从日志里算出来，需要额外推断或放弃。

### 11.5 对本项目核心目标的实际影响

如果把目标限定为“单局记牌训练”，当前启发式策略已经能满足大部分核心需求：

- 牌谱合法
- 节奏可读
- 题目可生成
- 无限新局

如果把目标升级为“更像真人实战复盘器”，那真实对局记录会明显更好。

所以当前最合理的判断是：

> 启发式牌谱适合作为 V1 的训练底座；真实对局记录更适合作为 V2/V3 的真实性增强层，而不是当前阶段的前置条件。

### 11.6 建议的演进路线

如果后续要继续做深，建议不是“全量替换”，而是混合方案：

1. 继续保留启发式生成，保证无限牌局和完整可控数据。
2. 新增少量真实牌谱专题训练，作为高阶模式。
3. 把真实牌谱用于校准启发式权重，而不是直接废掉现有引擎。

这样既能保住当前项目最强的可控性，也能逐步补足真实性。

## 12. 当前技术结构

### 12.1 前端

- React 19
- TypeScript
- Vite
- PWA manifest + service worker

### 12.2 引擎职责分层

- types.ts：定义牌、动作、轮次、快照、题目等数据结构。
- engine.ts：规则枚举、牌局生成、回放快照、剩余牌统计。
- challenges.ts：依据当前可见历史抽取问题。

### 12.3 前端页面职责

- App.tsx：主桌面、弹窗、控制、战报。
- PlayingCard.tsx：本地 SVG 牌面渲染。
- App.css / index.css：牌桌视觉、控制坞、弹窗和整体主题。

## 13. 部署方式

项目通过 GitHub Actions 在 push 到 main 后自动构建并部署到 GitHub Pages。Vite 的 base 会根据 GitHub Actions 环境自动切换，因此本地开发与 Pages 子路径部署可以共用同一套配置。
